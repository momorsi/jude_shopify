How should the web UI relate to the sync service?

The sync service currently runs as a Windows background process. Three ways to wire in a web UI.

A

Unified Process — FastAPI runs the sync engine

FastAPI becomes the main entry point. Sync tasks run as asyncio background tasks inside the same app. One process, one Windows service.

Windows Service → FastAPI app
├─ HTTP server (port 8000)
├─ orders_sync loop (asyncio task)
├─ inventory_sync loop (asyncio task)
└─ ... other sync tasks
Shared: SQLite event log + configurations.json

Pros

  • One process, one service to manage
  • Direct access to sync state in memory
  • Start/stop syncs from UI instantly

Cons

  • Bigger refactor of current entry point
  • UI crash = sync stops
B

Separate Processes — shared SQLite as the bridge

Sync service keeps running as-is. FastAPI web server is a second process. They communicate through a shared SQLite database: sync writes events, UI reads them and writes commands that sync polls for.

Windows Service 1 → continuous_main.py
└─ Reads commands from SQLite
└─ Writes events to SQLite
Windows Service 2 → FastAPI (web UI)
└─ Reads events from SQLite
└─ Writes commands to SQLite
Shared: sync_events.db + configurations.json

Pros

  • Sync survives if UI crashes
  • Minimal change to existing sync code
  • Clean separation of concerns

Cons

  • Two services to manage/start
  • Slight command latency (poll interval)
C

FastAPI wraps sync — UI triggers syncs directly via HTTP

FastAPI is the main process. Continuous sync loops run inside it. But syncs can also be triggered immediately via API endpoints. The existing continuous_main.py is replaced by the FastAPI app startup.

Windows Service → FastAPI app
├─ GET /dashboard
├─ POST /sync/orders → runs sync now
├─ POST /sync/stock?items=SKU1,SKU2
├─ Background: orders loop (asyncio)
└─ Background: inventory loop (asyncio)
SQLite: sync_events.db (event log)

Pros

  • One service, full control
  • Instant sync triggers (no polling)
  • Clean REST API for everything
  • Natural fit with asyncio already in use

Cons

  • Replaces continuous_main.py (moderate refactor)
  • Must handle graceful shutdown carefully
